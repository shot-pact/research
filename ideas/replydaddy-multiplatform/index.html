Design Doc · April 2026 · Idea: ReplyDaddy Multi-Platform

Multi-platform social reply co-pilot

Expanding ReplyDaddy from Reddit-only to Reddit + X + LinkedIn. Infrastructure, data model, AI pipeline, and unit economics for the extension.

01 · Scope

What we're building and what we're not

In scope

  • Discovery: monitor Reddit subreddits, X lists/keywords, and LinkedIn hashtags/companies for posts matching project keywords + intent.
  • Relevance ranking: embeddings + LLM rerank to surface only genuinely-engageable posts.
  • Reply generation: Claude-powered drafts in the user's brand voice, per-platform tone.
  • Posting: Reddit API and LinkedIn API for publishing; Chrome extension for LinkedIn engagement (comments/reactions) and X engagement fallback when API is rate-limited.
  • Multi-tenant projects: agencies can manage 5–20 client projects; solopreneurs get one.
  • Usage metering: credit-based per action; soft caps per tier.

Out of scope

  • Instagram, TikTok, Facebook Groups — no viable API path without TOS violation.
  • DM automation on any platform — legal grey zone and high ban risk.
  • Connection-request automation on LinkedIn (separate product category, high risk).
  • Fully autonomous replies without user approval. Users always confirm before a post ships.
02 · High-level architecture

System topology

Three concerns separated: platform adapters (any I/O with Reddit/X/LinkedIn), the AI pipeline (embeddings, rerank, generation), and the application core (projects, users, billing, dashboard). Each platform is an isolated module so an API change on one doesn't ripple.

Why this shape

  • Workers, not request handlers, do the expensive work. A discovery run pulls 500 posts, embeds them, reranks, and stores candidates — this can't block an HTTP request. BullMQ on Redis is boring and reliable; Temporal/inngest are fine alternatives.
  • Postgres + pgvector is the right first choice for everything except when you cross ~20M vectors. Do not start with Pinecone; you will burn $500/mo for nothing at early scale.
  • Chrome extension talks to backend over WebSocket, not just HTTPS, so the dashboard can tell the extension "go reply to candidate_post_id=123" in real time and get confirmation back.
03 · Platform adapters

Per-platform realities

Reddit Easiest

Reddit adapter

Read: OAuth with read scope, poll subreddit /new listings every 5–15 min. Free tier is 100 QPM per OAuth client; enough for hundreds of projects with per-project scheduling. Paid Reddit API ($0.24/1k calls) only needed at real scale.

Write: submit and edit scopes post comments/replies on behalf of the user. Must respect per-subreddit rules and rate limits (1 comment / 10 min per account is the safe default).

Risk: Reddit changed API pricing overnight in 2023 and killed Apollo. Treat all Reddit revenue as "API-pricing-exposed" and stash 20% of margin for that day.

X / Twitter Expensive API

X adapter

The trap: X API pricing post-Musk is brutal. Basic is $200/mo for 10K reads, Pro is $5,000/mo for 1M reads. A single scan pulling 500 tweets × 60 scans/month × 100 users = 3M reads, blowing past Pro.

Realistic hybrid: Use X API for user's own account (OAuth read + write their timeline, reply to specific tweet IDs). Use Apify's X scraper ($50–200/mo flat) for discovery/monitoring — much cheaper per-read, tolerates being slower.

Write path: Must use official API (scrapers can't post). OAuth 2.0 with tweet.write scope. Respect 300 tweets / 3-hour window per user.

Risk: Apify scrapers break whenever X changes DOM. Keep a second scraper provider on standby.

LinkedIn Hardest Highest ARPU

LinkedIn adapter · hybrid API + Chrome extension

Split the product into two paths — this is the single most important architectural decision for LinkedIn.

Publishing path (like Buffer)

Official LinkedIn Posts API. OAuth with w_member_social (personal) or w_organization_social (company pages). Fully sanctioned. Used for scheduled posts, article shares.

Apply to LinkedIn Marketing Developer Platform to get elevated quotas; partner status helps but isn't required for launch.

Engagement path (like Taplio)

Chrome extension injected into linkedin.com. Reads the user's feed via DOM, sends post candidates to backend over WebSocket. User clicks "reply" in the ReplyDaddy dashboard → extension fills the comment box → user confirms.

No API exists for this. Competitors all do it this way.

Chrome extension · architecture

Content script (DOM) ──► Background script ──► WebSocket to backend ▲ │ │ ▼ └── receives "reply to post X" command ◄── ReplyDaddy server │ ▼ Content script injects reply text User sees it, clicks send manually (or timed auto-send)

Critical safety rules (burned into the extension)

  • Max 25–40 actions per user per day, randomized.
  • Wait 30–180s between actions, with variance.
  • No activity outside user's typical waking hours (detected from past session times).
  • Stop immediately if LinkedIn shows a captcha, rate-limit page, or "unusual activity" warning.
  • Never two ReplyDaddy users sharing an IP without proxy rotation.

Risk: LinkedIn DOM changes break the extension every 6–10 weeks. Budget one engineer-week per quarter for maintenance. Users WILL get accounts restricted — have a clear incident-response doc and blame-shifting support policy, or you'll drown in tickets.

04 · Data model

Schema

Postgres as the source of truth. pgvector extension for embeddings. Tables are multi-tenant via org_id — everything cascades from an organization down.

Core tenant tables

-- Organizations: the billing unit. Solopreneur = 1 user, agency = many
CREATE TABLE organizations (
  id            uuid PRIMARY KEY,
  name          text NOT NULL,
  plan          text NOT NULL, -- 'solopreneur' | 'growing' | 'agency' | 'enterprise'
  stripe_customer_id text,
  created_at    timestamptz DEFAULT now()
);

CREATE TABLE users (
  id            uuid PRIMARY KEY,
  email         citext UNIQUE NOT NULL,
  name          text,
  created_at    timestamptz DEFAULT now()
);

CREATE TABLE org_members (
  org_id        uuid REFERENCES organizations(id) ON DELETE CASCADE,
  user_id       uuid REFERENCES users(id) ON DELETE CASCADE,
  role          text NOT NULL, -- 'owner' | 'admin' | 'member'
  PRIMARY KEY (org_id, user_id)
);

-- A project = one brand being promoted. Agencies have many.
CREATE TABLE projects (
  id            uuid PRIMARY KEY,
  org_id        uuid REFERENCES organizations(id) ON DELETE CASCADE,
  name          text NOT NULL,
  brand_voice   text, -- prompt fragment describing the voice
  target_icp    text, -- "SaaS founders, 20-50 employees, hate tool fatigue"
  website_url   text,
  product_desc  text, -- used in system prompt for all generation
  created_at    timestamptz DEFAULT now()
);

Platform accounts & targets

-- One row per connected social account. Tokens encrypted at rest.
CREATE TABLE platform_accounts (
  id            uuid PRIMARY KEY,
  project_id    uuid REFERENCES projects(id) ON DELETE CASCADE,
  platform      text NOT NULL, -- 'reddit' | 'x' | 'linkedin'
  external_id   text NOT NULL, -- u/username, @handle, linkedin URN
  access_token  text NOT NULL, -- encrypted
  refresh_token text,
  expires_at    timestamptz,
  scopes        text[],
  status        text DEFAULT 'active', -- 'active' | 'restricted' | 'revoked'
  health_meta   jsonb, -- karma, followers, last-scan-ok, warnings seen
  UNIQUE (project_id, platform, external_id)
);

-- What we watch. Unified shape across platforms.
CREATE TABLE targets (
  id            uuid PRIMARY KEY,
  project_id    uuid REFERENCES projects(id) ON DELETE CASCADE,
  platform      text NOT NULL,
  target_type   text NOT NULL, -- 'subreddit' | 'hashtag' | 'handle' | 'list' | 'company' | 'keyword'
  value         text NOT NULL, -- 'r/SaaS', '#buildinpublic', '@levelsio', etc.
  priority      int DEFAULT 50, -- 0-100, affects scan frequency
  is_exclude    boolean DEFAULT false
);

CREATE TABLE keywords (
  id            uuid PRIMARY KEY,
  project_id    uuid REFERENCES projects(id) ON DELETE CASCADE,
  text          text NOT NULL,
  intent        text, -- 'question' | 'complaint' | 'recommendation_ask' | null
  is_exclude    boolean DEFAULT false
);

Discovery & candidates (where the volume lives)

-- One row per scheduled or manual scan execution.
CREATE TABLE discovery_runs (
  id            uuid PRIMARY KEY,
  project_id    uuid REFERENCES projects(id),
  platform      text NOT NULL,
  started_at    timestamptz,
  completed_at  timestamptz,
  posts_scanned int,
  posts_matched int,
  cost_usd      numeric(10,4), -- sum of LLM + API costs for this run
  error         text
);

-- Every candidate post/tweet/linkedin-post the system has ever seen.
-- Partition by platform for query perf at scale.
CREATE TABLE candidate_posts (
  id              uuid PRIMARY KEY,
  project_id      uuid REFERENCES projects(id) ON DELETE CASCADE,
  discovery_run_id uuid REFERENCES discovery_runs(id),
  platform        text NOT NULL,
  external_id     text NOT NULL, -- reddit fullname, tweet id, linkedin URN
  author_handle   text,
  url             text,
  title           text,
  body            text,
  posted_at       timestamptz,
  engagement      jsonb, -- { upvotes, comments, likes, retweets } platform-dependent
  embedding       vector(1536), -- OpenAI text-embedding-3-small
  relevance_score numeric(4,3), -- 0.000 to 1.000, from LLM rerank
  intent          text, -- classified intent
  status          text DEFAULT 'new',
  -- 'new' | 'surfaced' | 'drafted' | 'replied' | 'dismissed' | 'stale'
  dismissed_reason text,
  created_at      timestamptz DEFAULT now(),
  UNIQUE (project_id, platform, external_id)
);

CREATE INDEX ON candidate_posts (project_id, status, relevance_score DESC);
CREATE INDEX ON candidate_posts USING ivfflat (embedding vector_cosine_ops);

Drafts & published actions

CREATE TABLE reply_drafts (
  id               uuid PRIMARY KEY,
  candidate_post_id uuid REFERENCES candidate_posts(id) ON DELETE CASCADE,
  project_id       uuid REFERENCES projects(id) ON DELETE CASCADE,

  persona_id       uuid REFERENCES personas(id),

  output_tokens    int,
  cached_tokens    int,
  cost_usd         numeric(10,5),
  status           text DEFAULT 'draft',
  -- 'draft' | 'approved' | 'scheduled' | 'sent' | 'failed' | 'deleted'
  scheduled_for    timestamptz,
  sent_at          timestamptz,
  external_reply_id text, -- the platform's ID for our posted reply
  error            text
);

CREATE TABLE scheduled_posts (
  id                uuid PRIMARY KEY,
  project_id        uuid REFERENCES projects(id) ON DELETE CASCADE,
  platform_account_id uuid REFERENCES platform_accounts(id),
  content           text NOT NULL,
  media_urls        text[],
  scheduled_for     timestamptz NOT NULL,
  status            text DEFAULT 'scheduled',
  sent_at           timestamptz,
  external_post_id  text,
  error             text
);

CREATE TABLE personas (
  id            uuid PRIMARY KEY,
  project_id    uuid REFERENCES projects(id) ON DELETE CASCADE,
  name          text,
  voice_prompt  text, -- "friendly builder, casual, no emoji, no buzzwords"
  example_replies text[], -- few-shot examples
  per_platform_hints jsonb -- { reddit: "...", linkedin: "..." }
);

Metering, credits, usage

-- Append-only usage log. Every billable action lands here.
CREATE TABLE usage_events (
  id            bigserial PRIMARY KEY,
  org_id        uuid NOT NULL,
  project_id    uuid,
  occurred_at   timestamptz DEFAULT now(),
  platform      text,
  action        text NOT NULL,
  -- 'scan' | 'embed' | 'rerank' | 'draft_reply' | 'draft_post' | 'publish'

);

CREATE TABLE usage_monthly (
  org_id        uuid,
  period_start  date,
  platform      text,
  action        text,
  total_credits int,
  total_cost    numeric(10,4),
  PRIMARY KEY (org_id, period_start, platform, action)
);
05 · AI pipeline

Relevance & generation

Every scan funnels 500+ raw posts down to 5–20 surfaced candidates. Cheap filtering first, expensive judgment last. This is the single biggest margin-control lever.

Raw posts (500) │ ▼ embedding lookup vs project intent vectors Candidate pool (150) ← cosine similarity > 0.65 │ ▼ cheap LLM intent classification (Haiku, tiny prompt) Intent-matched (40) ← is_question OR is_complaint OR is_rec_ask │ ▼ full rerank with product context (Haiku, prompt cached) Scored candidates (top 20) ← relevance_score > 0.75 │ ▼ surface in inbox for user review Surfaced (10-20) │ ▼ user clicks "draft reply" (Haiku default, Sonnet on toggle) Drafts │ ▼ user approves or edits, then schedule or post Published

Prompt caching — the single biggest cost lever

Brand voice + product description + ICP + persona examples run ~3,000 tokens and are identical for every reply in a project. Cache them. The 90% discount on cached input tokens cuts reply-generation cost roughly in half at volume.

// Per-reply payload, with cache breakpoint
{
  "model": "claude-haiku-4-5",
  "system": [
    { "type": "text",
      "text": PROJECT_BRAND_VOICE_AND_PRODUCT, // ~3000 tok, stable
      "cache_control": { "type": "ephemeral" } },
    { "type": "text",
      "text": PER_REQUEST_INSTRUCTIONS } // ~200 tok, fresh each time
  ],
  "messages": [{ "role": "user", "content": POST_CONTEXT_AND_TASK }]
}

Anti-spam guardrails (built into the generation prompt)

  • Never lead with the product. Answer the question first, mention product only if directly relevant.
  • No link unless the user explicitly enables it for that draft.
  • Per-platform tone: Reddit = casual/skeptical; LinkedIn = measured; X = punchy.
  • Flag to user if generated reply looks promotional (self-check pass via separate Haiku call).

Circuit breakers

  • If a platform_account sees a 429 or "account restricted" response, flip status to restricted and pause all queues for that account until user re-verifies.
  • Per-org daily cost cap: if daily LLM spend exceeds 3× the monthly-average daily cost, pause and alert ops — catches runaway abuse.
  • If Apify scraper failure rate > 10% in an hour, failover to backup scraper provider.

08 · Risks & mitigations

What breaks this

Platform policy

Reddit changes API pricing again. Keep a scraper fallback (Pushshift-like) warm. Budget 20% of Reddit-attributable revenue as policy reserve.

X API cost explosion

Basic → Pro is 25× cost jump. Hybrid with Apify already insulates; add a per-project X quota so a power user can't bankrupt the pool.

LinkedIn crackdown

Single biggest support cost. Ship conservative defaults, require user opt-in for the engagement extension, and clearly communicate "your account, your risk."

LLM abuse

Per-org daily cost circuit breaker (flag at 3× monthly-avg-daily). Credit-based pricing for anything beyond fair-use so cost is passed through.

Chrome extension decay

LinkedIn DOM changes every 6–10 weeks. Automated canary tests running in a headless browser against a dedicated test account catch breakage within hours.

Spammy reputation

Anti-spam self-check pass before drafts surface; kill accounts of customers posting obviously-promotional replies. Brand is the long-term moat.

09 · Build order

What to ship first

Week 1–2 · Foundation

Schema + auth + org/project/platform_account tables. Reddit adapter only. Embedding + Haiku rerank pipeline. Simple inbox UI. Ship this to existing ReplyDaddy Reddit users as a refactor.

Week 3–4 · X (API-only)

X adapter via official API only, no Apify yet. Reuse the rerank + draft pipeline. Offer as beta add-on to existing users for +$40/mo. Measure usage before committing to the Apify integration.

Week 5–8 · LinkedIn publishing (API)

Buffer-style LinkedIn posting via official API. Zero-risk, fully sanctioned. This alone unlocks the "multi-platform publishing" positioning and lets you charge $149+/mo. Do this BEFORE the engagement extension.

Week 9–14 · LinkedIn engagement extension

Chrome extension with WebSocket backend, conservative rate limits, clear user-facing risk disclosures. This is the expensive, risky work. Start with 50 beta users, learn the ban-rate, iterate.

Week 15+ · Agency tier

Multi-project UI, white-label, per-client reporting, consolidated billing. Priced at $799+. This is where the real revenue is — don't skip it, but don't build it until you have 3+ agencies asking for it.

10 · Open questions

Decisions to make before building

  • Credits vs unlimited. The current plan uses "unlimited replies**" with an asterisk. Moving to credit-based pricing protects margin but creates friction. Recommendation: keep "unlimited" for first-year marketing, but implement a hard fair-use ceiling (e.g. 1,500 replies/mo) silently.
  • Do we host the LLM or call APIs? Self-hosting Llama 3.3 70B on Together.ai or Groq would cut reply-gen cost by ~60%, but quality on brand-voice matching is measurably worse. Stay on Claude until ARR > $500k, then A/B test a hosted model for intent classification only.
  • Who owns the posted content? If a user gets sued over a reply generated by our tool, where does liability sit? Terms need an "advisory draft, user approves and owns" clause reviewed by a US lawyer.
  • LinkedIn account warm-up. Some competitors offer "account warming" where the extension slowly builds activity before doing real work. Powerful feature but materially increases ban risk for the user. Decide if we offer it at all.
  • Agency white-label scope. Full custom domain + logo, or just "Powered by ReplyDaddy" hidden? Full white-label typically demands enterprise pricing; decide price anchor.
